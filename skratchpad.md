# 📋 プロジェクト概要

## 🎯 目的

- 小規模タスクを進めるための、一時的PDCAメモ
- 粒度：数時間〜数日スパン、現在の作業だけ
- 更新頻度：手が動くたびに、常にアップデートすること！！
- 想定読者：今作業している自分＋AI
- サイズ上限：実務上 500 行程度でローテーション推奨
- 注意：モダンな実装よりも、初心者向きで管理しやすい簡易的実装を優先 (複雑なコードは削除せず、コメントアウトにす)

 ## 役割分担
- cursor:「UI/UX・フロントの体験」に集中
  - APIや型定義の変更が必要な場合はcursorにIssue/PRで依頼。
- windsurf:「API/DB/CI/CD/型定義」を主導し、
  - API仕様変更や新機能追加時はcusorに型・エンドポイント情報を即共有。
- o3:「GitHubのPR/Issue/README/外部OSS」を常時監視
  - 両者に「進捗レポート」「外部事例」「設計のヒント」を随時フィードバック。
  - 全体の進捗・課題・設計方針はskratchpad.mdやdevelopment_flow.mdで一元管理
  - ナレッジベースの構築：o3が開発中の学びやFAQを自動でまとめる
  - レビュー支援：o3がコードレビューの観点やテスト観点を自動でリストアップ

# 🎯 ダッシュボードへのタスク追加機能の実装手順

## 現状の問題点
- プロジェクト、タスク、サブタスクの各レベルで「ダッシュボードに追加」ボタンが機能していない
- 現在は`localStorage`に単一の`dashboardTask`として保存しているため、複数のタスクを追加できない
- タスクの状態管理が適切に行われていない

## 実装手順

### 1. データ構造の設計
- [ ] `DashboardTask`インターフェースの定義
  ```typescript
  interface DashboardTask {
    id: string;
    title: string;
    startTime: string;
    endTime: string;
    project: string;
    status: 'pending' | 'in_progress' | 'completed';
    createdAt: string;
  }
  ```

### 2. 状態管理の実装
- [ ] `useDashboardTasks`カスタムフックの作成
  - タスクの追加、更新、削除機能
  - ローカルストレージとの同期
  - タスクの並び替え機能

### 3. APIエンドポイントの実装
- [ ] バックエンドAPIの作成
  - `POST /api/dashboard/tasks` - タスクの追加
  - `GET /api/dashboard/tasks` - タスク一覧の取得
  - `PUT /api/dashboard/tasks/:id` - タスクの更新
  - `DELETE /api/dashboard/tasks/:id` - タスクの削除

### 4. フロントエンドの実装
- [ ] `project-template.tsx`の修正
  - `addToSchedule`関数の実装
  - `addSubtaskToSchedule`関数の実装

  - `addTaskGroupToSchedule`関数の実装

### 5. エラーハンドリング

  - APIリクエスト失敗時のエラーハンドリング
  - バリデーションエラーの処理
  - ユーザーへのフィードバック表示

### 6. テストの実装

### 補足：特に重視すること・・静的を動的にすることについて

特に、以下3点を非常に重視している。

- 1: プロジェクトの構造を動的にすること。今のtreeは静的になっている。最初は見本として3プロジェクトが入ってるのは良い。が、任意のプロジェクトを追加削除が「実際に」できる構造にまだなってない。

- 2: プロジェクトからダッシュボードへ、追加できるようにしてほしい。対象は、「プロジェクト/タスクグループ/タスク/サブタスク」を問わない。

- 3: プロジェクトページにおいて、AIタスク分解プロンプトが効くこと。(プロジェクト>タスクグループ>タスク>サブタスク)
